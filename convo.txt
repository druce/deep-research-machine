‚Źļ Good. Now let me propose approaches.

  Architecture Approaches

  Option A: FastAPI + WebSocket (Recommended)

  Backend: FastAPI serves the static HTML and provides:
  - GET /reports ‚ÄĒ scan work/ for completed runs with final_report.md
  - POST /run ‚ÄĒ launch research.py TICKER --date DATE as subprocess, return
  run_id
  - WebSocket /ws/{run_id} ‚ÄĒ tail the stream logs and push updates live
  - GET /status/{run_id} ‚ÄĒ query db.py status for task states + progress

  Frontend: Single index.html with Alpine.js. WebSocket for live log streaming +
   periodic status polling for task grid/progress bar. Auto-opens Typora
  server-side on completion.

  Pros: Real-time streaming, clean separation, FastAPI already in deps, proven
  pattern from v1.
  Cons: WebSocket adds slight complexity.

  Option B: SSE (Server-Sent Events) instead of WebSocket

  Same as A but uses SSE for log streaming instead of WebSocket. Simpler
  protocol (one-way), auto-reconnects.

  Pros: Simpler than WebSocket for one-way streaming.
  Cons: Can't easily send messages back to server. Less flexible if you later
  want bidirectional comms.

  Option C: Pure polling

  No WebSocket/SSE. Frontend polls /status/{run_id} and /logs/{run_id}?offset=N
  every 1-2 seconds.

  Pros: Simplest to implement, no connection management.
  Cons: Not truly real-time, more HTTP overhead, feels laggy.

  My recommendation: Option A (FastAPI + WebSocket). It's the proven pattern,
  gives genuinely real-time log output, and the dependencies are already
  installed. The previous implementation validated this architecture works well
  for this use case.
